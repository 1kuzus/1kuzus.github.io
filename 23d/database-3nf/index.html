数据库设计三大范式

null likes · null views · 2023-10-17

为了减少数据冗余，设计数据表时必须遵循一定的规则，在关系型数据库中这种规则就称为范式。
严谨的定义可以参考维基百科第一范式、第二范式、第三范式，互联网上也有很多优秀的技术博客。本文希望以简洁的语句和例子进行描述。

第一范式 - 1NF

数据库表的每一列都是不可再分的。

下面是一个不符合第一范式的例子：

销售部实习生

1004

陈宇航

吉林省长春市

销售部员工

其中的职务应该再分为部门和职务两个字段，因为业务中可能会经常分别用到这两个属性！

第二范式 - 2NF

符合第一范式，且非主键字段必须完全依赖于主键字段，不能只依赖主键的一部分。（主要针对联合主键而言）

下面是一个不符合第二范式的例子：

第三范式 - 3NF

符合第二范式，且非主键列都和主键直接相关，而不能间接相关（依赖于其他非主键字段）。

即第三范式要求不存在传递依赖，非主键属性之间应该是独立无关的。

下面是一个不符合第三范式的例子：

id

姓名

城市

城市人口/万

1

张三

重庆市

3100

2

李四

武汉市

1418

3

王五

北京市

2154

这个例子不符合第三范式，因为存在间接的传递决定关系：id → 城市 → 城市人口。城市人口字段直接依赖的是城市字段，而不是主键id。正确的做法是设计两张表：
